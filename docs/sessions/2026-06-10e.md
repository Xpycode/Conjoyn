  `@unchecked Sendable` idiom in `FFmpegWrapper`/`VerificationService`.
- **New `Services/TimecodeDisclosure.swift`** — display-only value `{sourceTimecode, appliedTimecode,

  `static async build(clips:settings:calendar:)`. `async` *only* for the `tmcd` read; the
  resolver/formatter work is sync. Crucially it re-runs the **same** `RecordingStartResolver` +
  `TimecodeFormatter` (same fallback fps = segment-1 probed or 30, same non-drop-frame) the engine

## Decisions
- **Disclosure is lazy/row-side, not frozen on the job** (deviation from the 2026-06-10c plan, user
  approved). Rationale: identical values from the same frozen inputs + same resolver, but far less
  surface area (no model field, no Codable migration, no async patch-back through `addJob`'s
  conflict-rebuild) and naturally per-row session-only state. Recompute cost is ~once per row.
- **`SourceTimecodeReader.Result` is `@unchecked Sendable`** — only safe-by-construction value types;
  needed to return it across the actor boundary under strict-concurrency `complete`.
- **No engine/export change** — Part 2 is honesty + exposing already-computed values; the join path
  is untouched, consistent with the 2026-06-09 "date+timecode stamp model" decision.
