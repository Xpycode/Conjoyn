### 2. Bar honesty — two symptoms, one root cause
Root cause: the row bar's colour + label were driven **only** by `job.status`, which has just
`.active`/`.completed`/`.failed` — collapsing the real five-phase lifecycle (join → staged-move
→ stitch → verify → done) so green fired the instant the file was *written*, and the move/stitch
tail ran under a hard-coded "Joining…".
- **`ConversionJob.isFinishing`** — new transient flag (out of `CodingKeys`, same proven pattern
  as `isDeepVerifying`); set in `processConcatenateJob` right after `mergeClips` returns; cleared
  on `.completed` / `.failed`.
- **`QueuePanel.statusText`** — `.active && isFinishing` → **"Finishing…"** (distinct from
  "Joining…" while ffmpeg runs, and "Verifying…").
- **`QueuePanel.barFill`** — green now **requires `verificationStatus == .verified`** (or
  `.warning`); `.verifying`/`.unverified` → orange; verify-`.failed` or `.status==.failed` → red.
  Principle: *green = verified byte-for-byte, never "the file exists".*

### 3. Byte-level move progress (the bar now climbs through the move)
- `moveIntoPlace` rewritten from `FileManager.moveItem` (no progress) to a **streamed copy**:
  8 MB chunks via `FileHandle`, reports `progress` 0…1, `synchronize()` (fsync) **before**
  deleting the source. Safety preserved: source dropped only after every byte is on disk → a
  mid-copy failure throws with `src` intact + partial `dest`, which the caller's existing
  `destinationTouched` cleanup deletes; original DJI segments are never touched (we copy the
  staged join, not the inputs). Falls back to atomic `moveItem` if source size is unreadable.
- **Composite bar (staged jobs):** `joinPortion = staged ? 0.5 : 1.0`. ffmpeg join fills
  `[0, 0.5]`, the cross-volume move fills `[0.5, 1.0]`. The 50/50 split is principled, not
  arbitrary — both phases shuttle ~the same byte count across the slow external link (join writes
  the joined file; the move copies it out), so equal bar-distance ≈ equal real work. Unstaged jobs
  have no move → join fills the whole bar.
- **Side benefit:** the active-job ETA (`remainingQueueSeconds` via `ProgressMetrics(progress…)`)
  now extrapolates over join **+** move instead of join-only — more honest, no regression to the
  per-batch session-pace pending estimate.
- **+1 test** `testMoveIntoPlaceStreamsProgressAndCopiesBytesIntact` (20 MB multi-chunk: bytes
  byte-identical, progress monotonic non-decreasing, ends at 1.0). Existing
  `testMoveIntoPlaceReplacesExistingDestination` still green through the new path.

## Part 2 (PM) — Timecode write-back verification (ported idea from TCE)

### Why
A `/status` aside asked whether TCE's "verification ladder" had anything Conjoyn could reuse.
Audit found Conjoyn already **equals or exceeds** TCE on the media-integrity dimension
(`SourceTargetVerifier` Tier 0 readability / Tier 1 packet-count·bytes·duration·codec / Tier 2
byte-exact `streamhash` — source↔output, stronger than TCE's single-file check). The **one** gap:
TCE's "Quick" level re-reads the output `tmcd` and confirms it matches the assigned timecode —
and Conjoyn's *headline* date/timecode fix was **stamped but never verified**.

### What shipped
- **`ConversionJob.appliedTimecode: String?`** (persisted, in `CodingKeys`) — records the exact
  string `resolveJoinMetadata` hands to ffmpeg's `-timecode`. Set in `QueueManager+Processing`
  right after resolve (`nil` when nothing stamped → check skips). Optional ⇒ old saved queues
  decode it as `nil` cleanly.
- **New check `VerificationCheck.Kind.timecodeWriteback`**, wired into `runTier0And1` so it runs in
  **both** fast and thorough tiers. Pure `compareTimecode(assigned:readBack:)` +
  `timecodeFields` (split on `:`/`;`) — **separator-insensitive** (drop-frame `;` vs non-drop `:`),
  field-by-field. "Proven wrong vs couldn't check" preserved: mismatch / missing-tag = `.fail`;
  unparseable / probe-failed = `.warning` (never a false fail).
- **`mapStatus` correctness fix:** the thorough-path "hash passed ⇒ verified" shortcut now
  **excludes** a `.timecodeWriteback` fail — a media hash can't speak to the `tmcd`, so a lost
  timecode still sinks the seal. (Without this, a byte-perfect join with a wrong tmcd would have
  read green.)
- **+6 unit tests** (exact / separator-insensitive / mismatch / missing-track / unparseable-is-warning
  / field-parsing). Suite **354 → 360 / 1 skip / 0 fail**.
- **Zero UI work** — the seal/chip row already iterates `result.checks` generically
  (`ForEach(…, id: \.kind)`); a passing check renders nothing, a flagged one shows a chip + tooltip.

### Live eyeball caught a real bug the units couldn't
First run on real `2CULL` footage flagged **⚠ Timecode write-back** on *both* jobs. Diagnosis via
`diagnostic.log`: the `tmcd` **was** written correctly (`timecode : 20:38:21:24`), but my read path
(`SourceTimecodeReader`, AVFoundation, ported from Penumbra) threw `.missingFormatDescription` —
an **ffmpeg-muxed tmcd attaches its format description to the track, not the sample buffer**
(camera-native files put it on the buffer). **Fix:** read the timecode via **ffprobe**
(`stream_tags=timecode`) — the same tool the rest of the verifier uses, immune to the AVFoundation
quirk; dropped the `SourceTimecodeReader` dependency from the verifier. Re-run → log shows
**"Timecode write-back: match"** ×5, Verify row reads **"No issues flagged."** with the ✓ seal.
*Lesson:* the pure/impure split paid off — the comparator never changed; the bug was isolated to
the swappable I/O method, exactly as designed.

(`CODE_SIGN_STYLE=Manual`, identity `…GREGOR MÜLLER (FDMSRXXN73)`) — this Mac has no "Mac
Development" cert (recurring; see 2026-06-17 eve). Binary mtime + version (1.0.2/102) confirmed
