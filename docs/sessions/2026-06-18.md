### 2. Bar honesty — two symptoms, one root cause
Root cause: the row bar's colour + label were driven **only** by `job.status`, which has just
`.active`/`.completed`/`.failed` — collapsing the real five-phase lifecycle (join → staged-move
→ stitch → verify → done) so green fired the instant the file was *written*, and the move/stitch
tail ran under a hard-coded "Joining…".
- **`ConversionJob.isFinishing`** — new transient flag (out of `CodingKeys`, same proven pattern
  as `isDeepVerifying`); set in `processConcatenateJob` right after `mergeClips` returns; cleared
  on `.completed` / `.failed`.
- **`QueuePanel.statusText`** — `.active && isFinishing` → **"Finishing…"** (distinct from
  "Joining…" while ffmpeg runs, and "Verifying…").
- **`QueuePanel.barFill`** — green now **requires `verificationStatus == .verified`** (or
  `.warning`); `.verifying`/`.unverified` → orange; verify-`.failed` or `.status==.failed` → red.
  Principle: *green = verified byte-for-byte, never "the file exists".*

### 3. Byte-level move progress (the bar now climbs through the move)
- `moveIntoPlace` rewritten from `FileManager.moveItem` (no progress) to a **streamed copy**:
  8 MB chunks via `FileHandle`, reports `progress` 0…1, `synchronize()` (fsync) **before**
  deleting the source. Safety preserved: source dropped only after every byte is on disk → a
  mid-copy failure throws with `src` intact + partial `dest`, which the caller's existing
  `destinationTouched` cleanup deletes; original DJI segments are never touched (we copy the
  staged join, not the inputs). Falls back to atomic `moveItem` if source size is unreadable.
- **Composite bar (staged jobs):** `joinPortion = staged ? 0.5 : 1.0`. ffmpeg join fills
  `[0, 0.5]`, the cross-volume move fills `[0.5, 1.0]`. The 50/50 split is principled, not
  arbitrary — both phases shuttle ~the same byte count across the slow external link (join writes
  the joined file; the move copies it out), so equal bar-distance ≈ equal real work. Unstaged jobs
  have no move → join fills the whole bar.
- **Side benefit:** the active-job ETA (`remainingQueueSeconds` via `ProgressMetrics(progress…)`)
  now extrapolates over join **+** move instead of join-only — more honest, no regression to the
  per-batch session-pace pending estimate.
- **+1 test** `testMoveIntoPlaceStreamsProgressAndCopiesBytesIntact` (20 MB multi-chunk: bytes
  byte-identical, progress monotonic non-decreasing, ends at 1.0). Existing
  `testMoveIntoPlaceReplacesExistingDestination` still green through the new path.

## Decisions
- **Green is an earned claim.** The success colour is tied to the *verification* result, not the
  I/O result — an app that promises lossless joins must never let the UI over-promise before the
  byte-exact check passes.
- **50/50 join/move split** for the staged composite bar — defensible by bytes-over-bottleneck,
  not a tuned guess. Documented inline.
- **Streamed copy over `moveItem`** accepted on the highest-stakes path *because* the
  fsync-before-delete ordering keeps the move atomic-in-effect (no data-loss window) while adding
  progress; validated by a byte-equality test before shipping behaviourally.

## Continuation (same day) — green-during-verify + one honest bar

Live eyeball of the move-progress build surfaced **two more honesty gaps**, both fixed and
eyeball-confirmed on real `2CULL` footage (two-job split run, then a 10-job singles run).

### 4. Green was leaking into the *verifying* phase (text + footer)
Yesterday's `barFill` fix made the **per-job bar** gate green on verification — but the **status
text** and the **footer outcome bar** still keyed on `job.status` alone, so a joined-but-still-
verifying file read **green in two places before it was verified** (green "Verifying…" text over
an orange bar; a green footer segment). Same bug class as #2, two surfaces it hadn't reached.
- **Root fix = one source of truth.** New **`VerificationStatus.outcomeTier`** (`verified` /
  `working` / `failed`) in `VerificationModels.swift`. `barFill`, `statusColor`, and the footer
  all now route through it, so "green = verified" is defined **once** and a future fourth surface
  can't drift. (`warning` counts as `verified`-green; `verifying`/`unverified` = `working`-amber;
  `failed` = red.)
- **`QueuePanel.statusColor`** — `.completed` now defers to the tier: verified → green (`ok`),
  working → amber (`acc1`, same as "Joining…" text), failed → red. No longer green-on-`.completed`.
- **Footer** (`outcomeSegments` + new `QueueManager.verifiedCount` /
  `awaitingVerificationCount` / `verifyFailedCount`): green segment = **verified** only;
  joined-but-verifying rides an **orange** (`acc2`) segment matching the per-job bar; verify
  failures fold into the red segment. `completedCount` (= "joined") = sum of the three tiers.
- **User choice:** "amber until verified" (reuse the working amber; green reserved for verified)
  over a dedicated 4th verify hue.
- **+1 test** `testCompletedJobsBucketByVerificationTier` (the three tiers partition completed jobs).

### 5. Verification folded into the single per-job bar
User: the main bar **sat full + orange and didn't move** during verify, while a *second* bar ticked
in the expanded detail — two "in progress" indicators, the main one looking stuck.
- **`ConversionJob.lifecycleFraction`** (+ `static producePortion = 0.85`) — one 0…1 fill across
  the whole lifecycle: produce (join+move, via `progress`) fills `[0, 0.85]`, verify
  (`verificationProgress`) fills `[0.85, 1]`. Produce **caps at 0.85 while active** so the
  hand-off into the verify slice is **continuous** (no backward jump when verify starts — safe
  because the `.completed`→`.verifying` flip happens in one synchronous main-actor stretch, so
  the full-then-drop frame never renders). Lives on the **model**, not the view, so it's testable.
- `QueuePanel` bar now reads `job.lifecycleFraction`; dead `barFraction` removed. The expanded
  detail keeps its own fine-grained verify bar (user chose "fold in **but keep** the detail bar").
- **Weighting decision:** rejected equal thirds (33/33/33). The three phases are wildly unequal and
  *conditional* — move is **0** when unstaged (temp==dest volume), and the fast auto-verify is
  seconds vs a ~40 s join — so equal thirds would *lurch* (move-third vanishes, verify-third snaps).
  Cost-weighted (join+move byte-split already in `progress`; verify = thin ~15 % tail; phase shown
  by the label) is the honest choice. User picked it over equal-thirds and over a visibly-segmented
  three-zone bar.
- **+1 test** `testLifecycleFractionFoldsVerifyIntoOneContinuousBar` (produce caps at
  `producePortion`; verify starts exactly there; full only at verified; failed = full track).

## Decisions (continuation)

  cost-weighting + a phase label tells the truth without a separate bar per process.
