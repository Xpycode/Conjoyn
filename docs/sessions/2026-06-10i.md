### 3. Live UI test surfaced 4 things (the productive part)
User drag-dropped the **card root** `/Volumes/M4P-1` (output `/Volumes/2CULL`) and saw "No video
segments found" — the clips live in `DCIM/DJI_001`. Diagnosed all four observations:
- **(a) Card-root drop finds nothing** → real gap (scan was `.skipsSubdirectoryDescendants`). **FIXED — see below.**
- **(b) "Queue doesn't clear" after killing the app** → **working as designed**: `QueueManager`
  persists `~/Library/Application Support/Conjoyn/queue.json` and reloads on launch (survives a
  crash/quit). The leftover `…_0006_D (1).mp4` was a restored prior job; `(1)` = collision-renamer.
  Use **Clear Queue**. No change.
- **(c) ETA wanted** → backlog. `SpeedTracker` already tracks throughput; surfacing a "~N min" readout
  is display-only, no new plumbing.
- **(d) "Read/write slower than before?"** → **not a regression, it's the SD card.** Source M4P-1 =
  `disk12` **Secure Digital**; dest 2CULL = `disk11` **USB** — different physical disks, so no
  same-disk contention. Activity Monitor showed **read ≈ 95 MB/s = the UHS-I SD-card ceiling** (the
  bottleneck). Earlier validated runs read from a **USB** source (`2CULL/2CULL-IN/DJI_001`), not the
  raw card — that's why it feels slower now. `+faststart`'s second pass (rewrites the file to front-load
  the moov atom) adds dest I/O near the end, more noticeable on a slow source. Card-bound, not app-bound.

### 4. Shipped card-aware folder descent (`feature/recursive-card-scan`)
