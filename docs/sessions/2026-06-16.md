# Same day — four UI changes shipped to `main` (all pushed, eyeballed live)

After the doc reconcile, ran a fresh HEAD build (`open-app-means-fresh-build`: xcodegen → clean
ad-hoc Debug build → launch, confirmed 1.0.1/101 not the stale DMG) and did four user-driven UI
changes. Each built clean (Swift 6), launched, user-confirmed. The `HelpMenu` SourceKit
"No such module" is a known false positive (vendored SPM pkg resolves at build; real build links it).

### 1 · File menu — split "Choose Folder…" into Source / Destination (`24d14b2`)
The lone **File › Choose Folder…** (⌘O) was actually the *source* picker but unlabelled, and the
destination picker had no menu entry (only the Output-bar path well). Renamed → **Choose Source
Folder…** (keep ⌘O), added a sibling **Choose Destination Folder…** → `chooseOutputFolder()` (so the
menu inherits that method's "re-point N pending jobs?" prompt for free), no shortcut. Wording matches
the path-well placeholders ("No source selected" / "No destination" — confirmed via the user's
screenshot, *not* the "Output" section header). One file: `ConjoynApp.swift` `FileCommands`.

### 2 · Appearance bug — Match System now reverts from a forced Dark/Light (`feb3c43`)
**Bug:** Dark → Match System stayed dark (Light/Dark toggling worked). **Root cause:**
`.preferredColorScheme(appearance.colorScheme)` with Match System → `nil`; on macOS
`.preferredColorScheme(nil)` does NOT clear a previously-forced `NSWindow.appearance`, so the window
stayed `darkAqua` and `Theme`'s dynamic `NSColor` tokens (resolve against the window's live
`NSAppearance`) kept returning dark. Dark→Light only worked because `.light` writes a concrete `aqua`
overriding the prior concrete value — the `nil` reset was the sole broken path. **Fix:** drive
**`NSApplication.shared.appearance`** (where `nil` reverts to system) via
`.onChange(of: appearance, initial: true)`; enum `colorScheme` → `nsAppearance` (`.aqua`/`.darkAqua`/
`nil`). → **Cookbook #113** (`113-macos-appearance-preference-nsapp-not-preferredcolorscheme.md`,
committed to the Directions repo `461ce75`, + Vestige). Reusable across every App-Shell sibling.

### 3 · App icon — new light variant + runtime light/dark Dock switch (`945ff4d`)
New design delivered as dark + light sets (`02_Design/conjoin-app-icon-dark-light.zip`; molten-weld
seam constant, only the surround flips). **Key finding (verified, not assumed):** macOS does **not**
honor an appearance-variant *bundle* icon — actool drops the dark renditions of an asset-catalog app
icon as "unassigned children" and ships a single appearance-agnostic `AppIcon.icns` (confirmed by
inspecting the built `Assets.car`: all renditions `appearance=(none)`, dark ones absent). The
HANDOFF's "Appearances: Any, Dark → swaps automatically" is **iOS-only advice**; on macOS the Dock
tile can only be re-skinned at runtime. So, per the user's follow-up ask, built a runtime switcher:
- `AppIconController` sets `NSApp.applicationIconImage` from a new `IconPreference`
  (Match System / Light / Dark); `.auto` tracks `NSApp.effectiveAppearance` via KVO so a System
  dark-mode flip *and* an in-app theme switch re-skin the running tile. Loads bundled
  `Resources/ConjoynIcon-{Dark,Light}.icns` (generated via `iconutil` from the delivered iconsets).
- `ConjoynApp`: `@AppStorage iconPreference` (default `.auto`) + `@StateObject` controller; `.task`
  starts it, `.onChange` applies menu picks.
- `AppearanceCommands`: theme radios → **Divider** → **"App Icon"** inline radio section.
- Bundle/Finder icon stays the **dark** catalog icon (byte-identical to what was already committed —
  the dark art was already in the repo; this delivery added the light variant). SVG masters + HANDOFF
  archived to `02_Design/app-icon/` as source of truth (supersedes the cookbook #76 CG generator).
- **macOS limit noted:** the runtime swap governs only the *running* Dock tile; Finder / not-running
  Dock always show the static bundle `.icns` — inherent, not a bug.
