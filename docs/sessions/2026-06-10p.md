## 1. Research — it was a wiring job, not new logic

Two Explore agents mapped (a) Conjoyn's existing speed/progress plumbing and (b) how the two
reference codebases surface ETA/speed. Key finding: **the ETA machinery was ported from P2toMXF and
is sitting unused.** Penumbra has no ETA at all, so P2toMXF (already inside Conjoyn) is the pattern.

Already present and tested, just not displayed:
- `ProgressMetrics.estimatedRemainingSeconds(at:)` (`ProgressModels.swift:34`) — the standard
  `elapsed / progress − elapsed` extrapolation, gated to ≥5% progress; `formattedRemaining(at:)` →
  `MM:SS` / `HH:MM:SS`.
- Every active job already carries `job.progress` (0–1, ffmpeg frame counting) + `job.startedAt`
  (`QueueManager+Processing.swift:42`); the row re-renders on each progress update.
- `queue.currentJobEstimate` (a `ConversionEstimate` with `formattedEstimate` like "~3 min") is
  computed at job start (`:46`) from SpeedTracker history — but nothing displayed it.
- ffmpeg's live `speed=` flows through the `metricsHandler` but was consumed only by the
  slow-speed check and thrown away.

The reference UI to copy: P2toMXF's `ProgressControlPanel` shows it with a
`TimelineView(.periodic by: 1.0)` + hourglass icon + `~` prefix.

## 2. Scope decided with the user

- **How far:** per-active-row ETA **+** a queue-wide footer total (for SD-card/watch-folder batches).
- **Speed too:** yes — show the live multiplier (`12.5×`) next to the ETA. This is the only choice
  that needed plumbing, since the live `ProgressMetrics` was being discarded.

## 3. Implementation (`a9f4de1`) — no new source files, no xcodegen regen

- **`QueueManager.swift`** — new `@Published var activeMetrics: ProgressMetrics?` (ffmpeg's live
  metrics for the active job).
- **`QueueManager+Processing.swift`** — the `metricsHandler` now stores `activeMetrics` alongside
  the existing slow-speed check; cleared at job start (`activeMetrics = nil`) and end. New testable
  `remainingQueueSeconds(at:)` = the active job's live `elapsed/progress` remaining (historical
  `currentJobEstimate` fallback before 5%) **+** `getTotalQueueEstimate()` for still-pending jobs.
  Returns `nil` when not processing.
- **`ProgressModels.swift`** — shared `formattedCoarseDuration(_:)` (`< 1 min` / `~N min` /
  `~Nh Nm`); `ConversionEstimate.formattedEstimate` now reuses it (de-duped).
- **`QueuePanel.swift`** — `QueueRow` shows `12.5× · ~2:34 left` inline (1 s `TimelineView` tick,
  active rows only; coarse historical fallback pre-5% so it's never blank while preparing).
  `FooterBar` shows the whole-queue `· ~N min left` next to the overall bar while processing.

**Design note:** the per-row ETA is **history-independent** (pure `elapsed/progress`), so the
countdown is robust on a fresh install with no speed records; only the pending-job portion of the
footer total leans on SpeedTracker history.

**Tests:** +5 in `QueueManagerTests` — `formattedCoarseDuration`, plus the four
