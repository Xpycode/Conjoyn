## 1. Window-minimum-size fix (`3928f47`)

**Symptom:** at the root `minWidth: 1000` floor the Output bar's middle row mangled — "Output"
clipped to "utput", "Timecode from recording time" wrapped to two lines, "Add to Queue" was cut off
the right edge. First reported 2026-06-10m with 3 reference screenshots in
`03_Screenshots/min-window-size_2026-06-10m/`.

**Root cause:** `.windowResizability(.contentMinSize)` is supposed to derive the window's minimum
from the content's intrinsic minimum — but it does **not** propagate the Output bar's true minimum
width through the intervening `VSplitView`. So the window could shrink to 1000 pt even though the bar
needs ~1160 pt, and the bar's elements truncated/wrapped/clipped instead of forcing a wider floor.

**First attempt (insufficient on its own):** `.fixedSize()` on the "Output" label + each switch
label. This stopped the *truncation/wrap*, but the window still shrank below the bar width — so the
bar then **overflowed and clipped on both edges** ("Output" off the left, "Add to" off the right,
"QUEUE"→"UEUE"). Confirmed via a user screenshot that `.contentMinSize` was not picking up the
fixedSize content through the VSplitView.

**Final fix:** measured the bar's intrinsic width with AppKit font metrics (a throwaway
`NSFont.systemFont` sizing script): ~1133 pt empty / **~1160 pt worst-case** (selected state, "Add 999
to Queue"). Set the root frame to `minWidth: 1220` (`ContentView.swift`) — measured worst-case plus a
comfortable margin — and kept the `.fixedSize()` labels as a backstop so nothing can ever wrap/clip at
the floor. The number is documented inline with the reasoning.

**Verified:** user dragged the window to its minimum — "Output" intact, "Timecode…" single-line,
"Add to Queue" fully visible, window stops at the comfortable width with no overflow. 238/238 tests
still green (display-only change).

## 4. False-alarm "crash" diagnosed

User reported the app "crashed" while selecting the `2CULL-IN` folder in the open dialog. Investigated:
the app **never died** — same PID alive for 10 minutes, zero crash reports for any process. It was a
**file-dialog beachball** entering `/Volumes/2CULL/2CULL-IN` (a USB APFS drive) — its `DJI_001`
subfolder has **349 entries** of large video, triggering a macOS thumbnail/enumeration storm,
compounded by the **boot disk at 96 % full** (~40 GB free). After the stall the Conjoyn window slipped
behind another app's window, so it *looked* gone. No Conjoyn code defect; not related to the
output-folder feature (the dialog is a stock `NSOpenPanel`; our code runs only after selection).
Flagged the near-full boot disk to the user.
