# 2026-06-25 (session 2, later) — Post-ship Q&A + website v1.0.3 badge

Short follow-up after the 1.0.3 ship. No app code changed; one website edit shipped.

## Goal
Answer a couple of architecture questions about the join output pipeline, surface (then defer) a
UX gap they exposed, and put the new version number on the public landing page.

## Progress
- **Traced the join output staging** (`QueueManager+Processing.swift`): the finished media is
  written to the **temp/scratch volume first** (default = boot SSD via `TempDirectoryManager`,
  user-redirectable) then copied to the destination — but **only when temp ≠ destination volume**
  (`staged = !DiskSpace.sameVolume(...)`, line 240). Same volume → writes straight to the
  destination, no copy. Rationale (comment 232-238): keep ffmpeg's write + the `+faststart`
  read-back-and-rewrite churn off slow external media, which otherwise threw the "re-open / I/O
  error" at the tail of long batches. The streaming move (`moveIntoPlace`) fsyncs before deleting
  the source, and the destination folder stays empty until the very end (`destinationTouched`).
- **Confirmed the disk-space preflight checks the temp volume** (`preflightDiskSpaceError`, 590):
  source size + 10% margin, each volume checked independently when staged, **2×** required when
  temp+output share one disk. Fails **open** on a `nil` capacity query (early-warning, not the last
  line of defense — the real write surfaces a genuine ENOSPC anyway).
- **UX gap found, then deferred by user:** both preflight error strings tell users to fix space via
  **"File → Temp Folder…"** — but **no such menu/picker exists**. The `TempDirectoryManager`
  backend (`setCustomTempDirectory`, UserDefaults persistence) is built + tested but never wired to
  any UI, so the scratch dir is always the boot volume. **User chose: leave as-is** (edge case;
  staging design itself is sound). The dead-end error text stays for now (known + accepted).
- **Website v1.0.3 badge.** Confirmed all 1.0.3 *deliverables* were already live (appcast, DMG,
  release notes, human `downloads/conjoyn.dmg` button — all from the ship session). But
  `index.html` carried **no version string at all** (version-agnostic by design). Added
  `<span>v1.0.3</span>` as the first chip in `.hero-meta` (reads "v1.0.3 · macOS 14.0+ · Apple
  Silicon · …"). Dry-ran the deploy (only `index.html` transferred — no DMG re-upload, no
  deletions), pushed live, **verified** `https://conjoyn.lucesumbrarum.com/` 200 + badge present.

## Decisions
- **Temp-folder picker UI — deferred (leave as-is).** Staging is well-justified; the missing
  "File → Temp Folder…" menu only bites when the boot SSD is critically full while joining to an
  external drive. The two preflight error strings reference a menu that doesn't exist — accepted
  known dead-end, not fixed this session.
- **Landing-page version — static hardcoded badge** over a dynamic appcast-fetch (simpler). Trade:
  it's now a **second source of truth to bump each release** (alongside appcast + DMG name +
  `downloads/conjoyn.dmg`). Noted in Next.
