# 2026-06-24 (later) ‚Äî Watch-folder daemon hardening (the 3 deferred engine-review items)

**Mac:** (this Mac) ¬∑ **Branch in/out:** `main` ‚Üí `fix/wave5-watchfolder-hardening` ‚Üí **`main`** (merged
`--no-ff` `2905b38`, branch deleted) ¬∑ **Code + tests changed; shipped 1.0.2/102 untouched.**

Closed the 3 worth-fixing items from the 2026-06-23 post-hoc engine review ‚Äî the hang / use-after-free /
time-of-check edge cases that the 5.14 happy-path eyeball couldn't exercise. Read every load-bearing file
first (the review pointed at line numbers; verified them against current source). One genuine product fork
surfaced (#1 recovery strategy) ‚Üí asked the user, who chose **timeout + keep cadence** (full self-heal).

- **#2 FSEvents teardown UAF** (`d7e05fe`) ‚Äî `WatchFolder`'s stream context used `passUnretained` + `nil`
  retain/release, so `stop()`/`deinit` on the owner's actor could free the monitor while a callback was in
  flight on the GCD queue. Gave the stream real **retain/release context callbacks** so it holds its own
  strong ref until `FSEventStreamRelease` (after `Invalidate`) balances it. This is an intentional
  stream‚Üîmonitor **cycle** broken by the explicit `stop()` the coordinator always calls ‚Äî audited the
  teardown (`deactivate ‚Üí disable ‚Üí stopMonitor ‚Üí stop()` on every remove/toggle/re-monitor) to confirm no
  un-stopped drop except at process exit (moot). `deinit` is now a fallback for the unstarted/already-stopped
  case. **Test gotcha learned:** `FSEventStreamRelease` runs the release callback **asynchronously** on the
  dispatch queue, so the dealloc lands a beat after `stop()` ‚Äî the lifetime tests had to *wait* for it, not
  assert synchronously. That async teardown is precisely what closes the UAF (the stream won't let `self`
  die until its queue drains). +3 lifetime/retain-balance tests.
- **#1 hung-discover deadlock** (`e3f9789`) ‚Äî `reconcile` latched one `isRescanning` flag cleared by
  `defer`; a `discover()` (ffprobe) that hangs forever never lets the `defer` fire, so the flag latched
  `true` and every later pass dropped ‚Üí watcher silently dead. **Lesson:** `defer` can't rescue a coroutine
  suspended forever at `await` ‚Äî it runs on scope *exit*, which never comes. Fix: split into `isDiscovering`
  (heavy rediscovery) + `isResampling` (cheap cadence) so a wedged scan can't freeze the cadence, AND bound
  `discover()` with `withDiscoverTimeout(seconds:)` (new tunable `WatchFolderSettings.discoverTimeout`,
  default 90 s). The wedged task is **abandoned, not awaited** (an ffprobe `Process` ignores cooperative
  cancellation; awaiting would re-deadlock) via a `withCheckedContinuation` race + single-resume `ResumeGate`
  actor. +1 test: a discover that wedges once then recovers proves the latch clears and the watcher enqueues.
- **#4 stale-key cache eviction** (folded into `e3f9789`) ‚Äî evict `sampleHistory`/`groupLastChanged` for
  clips/groups no longer discovered, bounding memory across a long daemon session over many cards. Pure
  caches only; persisted `groupStates` untouched. Pruned only when a *discovery* confirms a group is gone
  (poll passes reuse `lastGroups`, so a transient read blip evicts nothing).
- **#3 enqueue‚Üíjoin TOCTOU** (`3ee5933`, cookbook #127) ‚Äî clips captured at enqueue, ffmpeg runs minutes
  later; a card swap / in-camera rotation repoints a `DJI_NNNN` path at different bytes that ffmpeg would
  join silently. New `FileIdentity` snapshots `(device, inode)` via `lstat` at enqueue (in the shared
  `addJob` funnel ‚Üí manual queue guarded too) and re-verifies before `mergeClips`; `.mismatch`/`.missingNow`
  throws the **non-retriable** `FFmpegError.sourceIdentityChanged`. **Policy call (logged in decisions.md):**
  `.unverifiable` (transient `stat` error ‚â† "gone") does *not* block ‚Äî #127's source was *trashing* (refuse
  on uncertainty is safe there), this is *read-and-produce*, so a momentary read blip shouldn't fail a legit
  job (ffmpeg surfaces a real read error itself). Restored job = no baseline = no-op. Identities transient
  (out of `CodingKeys`). +9 tests (FileIdentity capture/verify + job snapshot/recheck + addJob captures).
