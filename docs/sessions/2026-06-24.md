**First run — PASSED.** Added `/Volumes/M4P-1` as a watch folder (output to an internal Desktop
folder, off-card). **6 DJI groups** discovered → joined `-c copy` → SRT-stitched → fast-verified;
the 4-segment set stitched to a single **13.46 GB** file. `=== Queue processing complete ===`,
**0 failures**, timecode write-back matched, card originals (9 source `.MP4`) untouched.

**TCC finding — the prompt never fired, and that's correct.** Picking the folder through the
`NSOpenPanel` ("Add Folder") is the **powerbox**: macOS treats it as explicit consent and grants
access to that path directly, with **no `NSRemovableVolumesUsageDescription` prompt** even on a
removable volume. That key gates *programmatic / background* access, not user-selected paths —
`WatchFolderBookmark.swift` documents exactly this. So no prompt on add = expected.

**Relaunch — PASSED.** Quit fully (Command-Q) → relaunched the same build. The watch entry + plain
bookmark + `ProcessedGroupLedger` (6 fingerprints, persisted under
`Application Support/Conjoyn/WatchFolders/<uuid>/processed_groups.json`) all survived:
- (a) **no permission re-prompt** — the grant persisted across relaunch for the background read;
- (b) watch **resumed from the bookmark** without re-picking (the panel window is summoned from the
  menu by design, but the entry was active);
- (c) **queue stayed empty — no re-join** (the dedup ledger held; `diagnostic.log` grew 0 lines,
  0 new ffmpeg joins, 0 `EPERM`).

**Live-detection drop-test — PASSED (the decisive one).** To prove the relaunched watcher is
*genuinely live* (not just UI-restored) and that **background** card reads work — the real concern
behind the engine review's "silent watcher death" finding — copied a source clip onto the card
under a fresh index (`DJI_20260524021500_0099_D.MP4`+`.SRT`). With no panel pick and no prompt, the
background watcher **auto-detected it in ~25 s** (FSEvents → stability gate → quiet window →
enqueue → ffmpeg) and joined it end-to-end (**17.4× realtime**, all checks passed). Bonus: the
date/TC discrepancy handler fired correctly — the renamed file's `05-24` filename vs the copied
SRT's `05-21 17:47:15` cue logged `⚠︎ SRT and filename disagree by 196065 s — using SRT first cue`,
exactly the "timecode/SRT authoritative, surface the discrepancy" decision (the mismatch is a test
artifact; real footage agrees).

**Why no prompt is the *answer*, with one honest caveat:** the original 5.14 open question was
"does macOS keep granting the read across relaunch?" — answered **yes**, with zero permission
friction. Because the dialog never visually fired, we can't fully distinguish "powerbox grant
persisted" from "the built-in SDXC slot isn't TCC-gated like a USB SD reader would be." For this
hardware the practical outcome is identical, and the drop-test proves background programmatic reads
work regardless. A USB SD reader (different bus) might still surface the dialog — untested.
