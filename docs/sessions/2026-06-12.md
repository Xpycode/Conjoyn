# 2026-06-12 (b) — Sortable recordings columns (backlog 12, → v1.0.1)

**Focus:** Finder-style clickable column headers for the discovered-recordings list. First post-1.0
feature.

## Decision: UI style
- User chose **clickable column headers** over a "Sort by ▾" menu. Sort keys: **Name, Date,
  Duration, Size** (Files left as a static label — not a chosen key).
- Confirmed this ships as **v1.0.1** (the app is already a notarized DMG; this is post-release).

## Implementation
**`ConversionViewModel.swift`**
- New `SortKey { found, name, date, duration, size }` + `@Published sortKey` / `sortAscending`.
- `setSort(_:)` — Finder semantics: re-click active column flips direction; switching columns adopts
  that column's `defaultAscending` (Name A→Z; Date/Duration/Size lead descending = newest/longest/
  biggest first). `resetSort()` on rescan.
- Sorting folded into the existing `filteredGroups` chokepoint (filter → sort), **decorate-sort-
  undecorate** so each date resolves once, not per comparison. `groups`, selection, rename cache
  untouched — only the view feed reorders.
- Comparator extracted as a **pure static `orders(_:before:by:)` over a plain `SortField`** value —
  that's what made it unit-testable without a scanned card. Stable `groupIndex` tie-break so equal
  rows never jiggle; `localizedStandardCompare` for numeric-aware name order.

**`Views/RecordingsList.swift`**
- New `ColumnHeaderBar` + `SortLabel`; shared `CJRowMetrics` widths so the header's FILES·DURATION·
  SIZE pixel-align over the rows (NAME·DATE share the left region — date is a row sub-line, not its
  own column). Active column shows an up/down accent chevron.

**Tests:** +11 in `ConjoynTests/RecordingSortTests.swift` (direction state machine, numeric name
order, tie-break stability, nil-date policy). **Full suite 330 pass, 0 failures** (was 319-ish).

## Two bugs caught live (eyeball iteration)
1. **Header ballooned vertically** — used `Color.clear.frame(width:)` for leading slots; `Color.clear`
   is **greedy in any unconstrained axis**, so it stretched the bar to fill all height and centered
   the labels in a tall band. Fix: padding-based leading inset (`CJRowMetrics.leadingInset`), no
   greedy spacers → compact bar.
2. **"DURATION" wrapped to two lines** — label wider than its 66 pt column. Widened `durationCol`
   66→80 (shared by header + rows so they stay aligned) + `.lineLimit(1)` guard.

## Bug caught live: resort beachball
- Symptom: re-sorting hung ~1s while thumbnails were still loading. User blamed thumbnails.
- **Real cause:** the date sort resolved recording-start dates **synchronously inside
  `filteredGroups`** — which runs in view-body on the main thread — opening up to 74 `.SRT` files
  **off the SD card**. Thumbnails (an `actor` + async FFmpeg) were never the blocker; the cold cache
  just coincided with thumbnail loading.
- **Fix (two parts):** (a) `sortField` now reads only the *already-resolved* cache, falling back to
  the in-memory embedded date — a sort is pure memory, can't beachball. (b) `prewarmStartDates()`
  resolves all dates **off-thread** (`Task.detached`, `.utility`) right after scan, passing only
  `Sendable` values (`[(UUID, DJIClip?)]` + `Date?`), then bumps `@Published startDateRevision` so
  the list re-sorts into corrected order once warm. Synchronous `resolvedStartDate(for:)` kept for
  the rename path (user-triggered, selected-only). User confirmed "works fantastic."

## Decisions
- nil-date sort policy: **`nil → .distantPast`** (undated rows cluster at one end, flip with
  direction). Flagged the Finder "always-last" alternative as a `TODO(you):` in `orders(...)` —
  left at the default for now.

# 2026-06-12 (d) — Sparkle auto-update research (pre-planning, no code)

**Focus:** Decide nothing, build nothing — reconnoiter how every shipped sibling app wired up
Sparkle so next session's *planning* starts fully informed. Multi-mac pre-flight clean
(`## main`, 0/0, `origin/main` `4244ab2`).

## Method
- Fanned out **3 parallel research agents** (read-only) over the sibling apps:
  - **Penumbra** — closest living sibling (lucesumbrarum vendor, self-hosted appcast, bundled
    ffmpeg/ffprobe, hardened-runtime). Has `docs/PLAN-Sparkle.md` + `docs/sparkle-release-runbook.md`.
  - **CropBatch + P2toMXF** — P2toMXF is Conjoyn's literal port ancestor (same FFmpeg/non-sandbox
    profile); CropBatch is published with a dedicated `sparkle-signing.md`.
  - **Cookbook #16 + AppUpdater + the smaller impls** (DiskVerdict, QuickMotion, AspectRatioUnifier,
    syncthingStatus) — to extract the house-standard pattern and variations.
- Consolidated everything into **`docs/sparkle-research.md`** (the deliverable; feeds next session).

## Key findings
- **Conjoyn is the *simplest* Sparkle case in the family** — because it's **non-sandboxed**.
  Almost every sibling scar is sandbox-specific and Conjoyn sidesteps it: **no new entitlement,
  no new build setting**. The existing FFmpeg entitlements (`cs.disable-library-validation` +
  `allow-unsigned-executable-memory` + `allow-jit`) are *exactly* what Sparkle's XPC needs too.
  Drop `network.client`, `SUEnableInstallerLauncherService`, and the `-spks`/`-spki` mach-lookup
  exceptions — all sandbox-only (Penumbra-only).
- **P2toMXF code is copy-verbatim** (`UpdaterController.swift` + `FileMenuCommands`/`@StateObject`
  menu pattern) — but P2toMXF **never finished**: its `SUPublicEDKey` is the placeholder
  `PENDING_GENERATE_KEYS_STEP` and it shipped no real appcast. So reuse its *code*, but take the
  *process* (signing/hosting/runbook) from **Penumbra**, which actually wrote it.
- **xcodegen wrinkle:** siblings add Sparkle via the Xcode GUI (pbxproj). Conjoyn must declare the
  SPM dep in `project.yml` and put the `SU*` keys in the **base Info.plist** — `INFOPLIST_KEY_*`
  silently drops custom keys (same trap class as cookbook #89's `UIDesignRequiresCompatibility`).
- `AppUpdater/` is **not relevant** — it's a standalone MacUpdater-replacement that *reads* other
  apps' feeds, the opposite side of the relationship.

## Landmines catalogued (each actually hit a sibling)
1. **EdDSA key loss = users can never auto-update again** (happened: SyncthingStatus). Generate a
   **fresh** Conjoyn key, back it up to 2 out-of-repo spots before first signed release, and bake
   `--account conjoyn` into every `sign_update`/`generate_appcast`.
2. **Never `codesign --deep`** (corrupts Sparkle XPC sigs) — Archive→Export(developer-id) instead.
3. **Notarization inspects Sparkle's nested binaries** (Autoupdate/Updater.app/Installer.xpc/…) —
   each must be Dev-ID + hardened-runtime, same as ffmpeg/ffprobe. Sign script must **leave
   Sparkle.framework alone** (only re-sign ffmpeg/ffprobe).
4. **`sparkle:version` = monotonic integer `CFBundleVersion`** (CropBatch shipped a downgrade loop).
5. **Enclosure `length` = exact bytes, no gzip/CDN recompression** or EdDSA verify breaks.
6. **First release can't test itself** — needs a throwaway 999→1000 staging test over local HTTPS.

## Hosting note
- Self-host like Penumbra (`conjoyn.lucesumbrarum.com/appcast.xml`). Strato: `lftp mirror -R`
  **without `--delete`** (protects the PHP download-counter state); point Sparkle's enclosure at the
  **raw DMG path**, not the counted/302 PHP endpoint.

## Open decisions left for the planning session (deliberately undecided)
- **DMG vs ZIP enclosure** (precedent favors ZIP for deltas; PROJECT_STATE says DMG — both work).
- Feed URL shape, Sparkle version pin, where `sign_update`/`generate_appcast` slot into
  `make-dmg.sh`, automatic-vs-manual checks.

## Next
- **Plan the Sparkle integration** from `docs/sparkle-research.md` (the whole point of this recon).
- Then the unchanged backlog: website copy + download link, QL thumbnail fix, nil-date sort policy.
