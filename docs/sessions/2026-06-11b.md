### Deadlock caught + fixed (autonomous bug-hunt) — `cf3de85`
A post-implementation review of the verifier found a real defect the test suite couldn't catch: the
process runner read ffprobe stdout from a **`Pipe` in the termination handler**. ffprobe's per-packet
`-show_entries packet=size` output is one line per packet — **hundreds of KB on a real multi-segment
join**, far past the ~64 KB OS pipe buffer — so the child would block writing, never exit, and the
handler would never fire: **the queue would hang after every real join.** The tiny test clips
(~600 bytes) never reached the threshold. Two concurrent-drain attempts made it *worse* (hung even on
tiny output — `readDataToEndOfFile` never saw EOF off Foundation's run-loop machinery), diagnosed
definitively by **`sample`-ing the stuck process** (blocked at `runCapturingStdout`). Final fix:
**capture stdout to a temp file** — the child writes to disk and never blocks, so reading it in the
termination handler (the mechanism `VerificationService.runProcess` already uses) is correct at any
size. The integration test that hung now completes in ~1.2 s.

## Decisions
- Verification is **true source↔target**, not decode-only: exact container-index comparison by
  default (cheap, automatic after every join), byte-exact packet-MD5 hash on demand or auto-escalated
  on a Tier-1 anomaly. The old decode-only `VerificationService` is left in place but **unwired** (its
  `passesFrameCheck` calibration tests still reference it); `deleteOriginalsAfterVerify` gating on the
  new result is a noted follow-up.
- All ffmpeg/ffprobe `-map`/`-select_streams` restricted to **kept streams** (`v:0`/`a:0`) so the
  dropped mjpeg-preview/DATA tracks never pollute the comparison.
- **Subprocess stdout is captured to a temp file, not a pipe** — pipes deadlock on ffprobe's
  per-packet output volume. (Codebook-worthy: the `Pipe` + terminationHandler idiom only survives
  small output.)
