## Decisions
- **Rename state on the ViewModel, not `ConversionSettings`** (reaffirmed from 2026-06-10c): the
  pattern is session-only; `ConversionSettings` is `Codable` and frozen onto every job, and only the
  *resolved* output name needs to ride on the job (it already does via `destinationURL`).
- **`{date}`/`{time}` derive from the same `RecordingStartResolver` instant** as the date/TC stamp,
  so the filename and embedded metadata can never disagree. Memoised per group.
- **Rename OFF = unchanged** — patterned naming is a separate branch in `addToQueue`; the default
  namer (and its `(1)` on-disk resolver) is untouched, so this side-steps (does not fix) the doubled
  `_D_D` variant-suffix bug, per spec.
- **Unresolved date → empty tokens** (engine), relying on the empty→`{name}` fallback for the fully
  empty case. Documented + tested (leaves a trailing separator in mixed patterns; acceptable edge).
- **VSplitView over `layoutPriority`** for the list/queue split — hands the ratio to the user and
  persists their drag; seeds nearer 50/50 (which is the bigger queue the user wanted anyway).
- **Clear Queue keeps the active job** — mirrors the existing `removeJob` guard; protects the open
  ffmpeg file handle + `currentJobId`/bookmark from being orphaned mid-write.
