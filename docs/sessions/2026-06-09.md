**`passesFrameCheck(decoded:expected:)` — implemented and merged** (was the one deliberate stub).
The full-decode pass criterion, the load-bearing quality judgement of verification (trust the
stitched file vs. flag it). Three cases, each chosen to fail *safe*:
- **Normal (usable estimate):** `decoded >= expected * 0.95` — a 5% shortfall tolerance. Catches
  the classic truncated-join failure (a last segment cut short loses far more than 5%) while
  tolerating the frame-or-two wobble of a `duration × fps` estimate off rounded metadata / VFR.
- **Over-count** (`decoded > expected`, common with B-frames/rounding) → passes trivially.
- **No usable estimate** (`expected <= 0`, incl. defensive negatives — ffprobe couldn't size the
  file): pass iff `decoded > 0`. Auto-pass would hide truncation; auto-fail would nag on every
  unsizable file.
The 5% is P2toMXF's proven default, extracted to `VerificationService.frameShortfallTolerance`
and **flagged for calibration against real DJI footage (Wave 6)** — the "right" value is empirical.

## 1.10 ThumbnailManager (`3afbae3`, 146 tests)
Ported `ThumbnailManager` + `FFmpegWrapper+Thumbnails` from P2toMXF on `feature/wave1-thumbnailmanager`,
merged `--no-ff` to `main`.

- **Ports verbatim:** the actor's LRU cache (100-entry, access-order eviction), the bounded-concurrency
  semaphore (max 3 concurrent ffmpeg processes, with the cancellation-aware continuation handoff), and
  per-request/`clearCache` cancellation. All format-agnostic.
- **DJI change:** P2 clips had a per-frame PROXY→ICON→VIDEO(MXF) fallback chain (a P2 card carries
  low-res proxies + a first-frame BMP). A DJI segment is a single self-contained MP4/MOV, so
  `extractThumbnails` reads the first frame at 0 s and the last frame near the end **directly from
  `clip.videoURL`** — no fallback chain, and the P2 `ThumbnailSource` indicator enum is dropped.
- **`DJIClip.lastFrameSeekSeconds`** (new, mirrors P2's `lastFrameTimestamp`): seek backed off
  `max(2/fps, 50 ms)` from the end. The bug the integration test caught: `duration − 1 frame` = 1.9667 s
  on a 2 s/30 fps clip, but `extractFrame` formats the seek as `%.3f` → `"1.967"`, which rounds *past*
  the final frame's PTS (and the container duration is itself an upper bound) → ffmpeg decoded nothing
  and the last-frame thumbnail came back nil. The wider guard lands reliably inside the file; the visual
  difference for a thumbnail is imperceptible.
- **`VideoStreamParams.framesPerSecond`** (new, additive): reusable `"num/den"` rational decode, `nil`
  for `"0/0"`/malformed (so `lastFrameSeekSeconds` falls back to 30 fps, never NaN/inf).
- **Tests (11):** rational decode (NTSC/integer/unknown), seek arithmetic (probed fps, 30-fps default,
  50 ms floor at 120 fps, zero-clamp for sub-frame clips), cache lifecycle (caches even an empty result,
  `clearCache` empties it) without ffmpeg, + one skippable real-ffmpeg test that generated a 2 s clip and

# Session 3 — 2026-06-09 — Exploratory research: multi-brand file-splitting survey

## Focus
Exploratory (no code). For the post-DJI multi-camera roadmap (memory `multi-camera-future`):
**which camera systems split their recordings, why, how the segments are named/ordered, and what a
pluggable parser/grouper would need per family.** Four parallel research agents (action cams /
mirrorless+DSLR+camcorders / drones+gimbals+FPV / split root-cause+tooling) + web verification.

## What happened
Wrote the full findings to **`docs/camera-split-research.md`** (brand-by-brand tables + sources).
Headlines:

- **Universal cause:** FAT32 4 GB-per-file limit (SD ≤32 GB = FAT32 = splits; ≥64 GB = exFAT). But
  **DJI/Autel/Canon split at ~4 GB even on exFAT** (firmware crash-safety cap). The 29:59 limit is a
  separate tariff mechanism that *stops* recording — it does **not** create a continuation file.
- **Two families.** **GoPro is the only brand that encodes continuity in the filename** (`GH01####` /
  `GH02####` — trailing 4 digits = constant recording ID, middle 2 = chapter). **Everyone else**
  (DJI, Osmo, Autel, Skydio, Parrot, HoverAir, Sony, Canon, Nikon, Panasonic, Fujifilm) just
  **increments a counter** → filename can't prove continuity → must group by metadata. **This
  validates our core "group by metadata, not filenames" decision.**
- **Our existing decisions all confirmed by the research:** metadata-continuity grouping, the
  camera-variant no-merge guard (parallel sensors = different codec/res), the slow-mo dual-timebase
  trap (use real elapsed time / size-cap, not container duration), and `-c copy` + `+genpts` +
  faststart with a TS-remux fallback for non-monotonic-DTS.
- **Pluggable seam is small:** only the **filename parser** (ordering + variant suffix) and an
  optional **AVCHD `.MPL` playlist reader** need to be per-brand. The grouper, the
  **stream-parameter identity hard gate**, the join engine, queue, verification, and sidecar
  stitching all stay shared. Adding GoPro ≈ a filename parser + carrying its embedded GPMF telemetry.
- **AVCHD is legacy** — Sony/Panasonic moved to MP4-wrapped XAVC-S since ~2014; AVCHD survives only
  as a secondary mode on old camcorders. It's also the one structurally-different format (`.MTS`
  transport stream in `PRIVATE/AVCHD/BDMV/STREAM/` + `.MPL` playlist). **Decision: defer / "only if
  asked."**
- **Product gap is real:** auto-detect joiners (ReelSteady, Gyroflow) are GoPro-only; the
  cross-format joiner (LosslessCut) is fully manual. **Nobody does auto-detect + multi-brand +
  lossless + telemetry preservation + watch-folder ingest** — exactly our niche.

## Decisions
- **Phase-2 target = the modern MP4/MOV family** (GoPro first, then DJI Osmo, Sony XAVC-S, Canon,
  Nikon, Panasonic, Fujifilm). ~90%+ of footage shot today; all reuse the existing engine + a
  per-brand filename parser.
- **Defer AVCHD** (legacy + structurally different — `.MTS`/`.MPL`, highest effort-to-relevance).
  360 reconstruction (proprietary) and Blackmagic (single-file by design) also deferred / no-op.
- **Stream-parameter identity is a hard gate before any join, for every brand** — the single biggest
  correctness lever; refuse/warn on mismatch rather than emit a corrupt `-c copy`.

## Next
- No immediate action — this is reference material for when the multi-camera rebrand starts (gated on
  DJI being validated on real footage first). When that begins: GoPro filename parser + GPMF
  telemetry path is the natural first slice.
- Carryover from Session 2 unchanged (drive the real app on the card end-to-end; GPL→LGPL FFmpeg swap
  before release).

# Session 4 — 2026-06-09 — Swap interim GPL FFmpeg for a from-source static LGPL build (task 6.1)

## Focus
Clear the last licensing release-blocker: the bundled `ffmpeg`/`ffprobe` were OSXExperts 8.1
`--enable-gpl` builds (x264/x265/…) — GPL source-distribution burden, MAS-incompatible. A
copy-only joiner needs none of the GPL encoders, and a plain FFmpeg build is LGPL by default.

## What happened

### Verified scope before building
Confirmed the app only ever invokes copy/metadata operations — concat demuxer, `-c copy`,
`-map 0:v:0`/`0:a?`/`-0:d`, `-movflags +faststart`, `-metadata`, `-timecode`, ffprobe JSON
(`-show_format`/`-show_streams`), mjpeg thumbnails (`scale` filter + native mjpeg encoder), and
`-f null` decode-verify. **All are in a default (LGPL) FFmpeg** — zero GPL components, zero
external libraries required.

### New reproducible build script — `01_Project/scripts/build-ffmpeg-lgpl.sh`
Builds `ffmpeg` + `ffprobe` from **unmodified FFmpeg 8.1 source** (same version as the interim
binary, tarball from ffmpeg.org):
- `--enable-static --disable-shared`, **no** `--enable-gpl`, **no** `--enable-nonfree` → license
  defaults to **LGPL v2.1+**.
- `--disable-network --disable-autodetect` → hermetic: won't link any stray Homebrew GPL libs that
  happen to be installed (x264/x265). Smaller attack surface for notarization.
- Kept the full built-in codec/demuxer/muxer set (no `--disable-everything`) so no DJI container
  quirk can break a join.
- **Hard gates the output**: the script `exit 1`s if either binary reports `--enable-gpl` /
  `--enable-nonfree` or isn't arm64. No nasm needed (arm64 NEON `.S` is assembled by clang).

### Result + validation
- `ffmpeg -L` reports **LGPL**; config string carries no `--enable-gpl`/`--enable-nonfree`.
- `otool -L` shows **only** `/usr/lib/libSystem` + Apple system frameworks — no `/opt/homebrew`,
  no `libx264`/`libx265`, no `/usr/local`. Self-contained single Mach-O each (so the static plan's
  promise holds: no `install_name_tool` dylib dance for these).
- **20 MB each, down from 52 MB** (GPL build carried all the external encoders).
- **Real-footage test** (card `DJI_001`, `0003_D` + `0004_D`): ffprobe JSON read OK (hevc + DJI
  data + mjpeg-preview streams enumerated); concat `-c copy` join with the app's exact flags exits

# Session 6 — 2026-06-09 — Date+timecode stamp model decided (unblocks task 2.8)

## Focus
No code. Resolve how the joined file should carry its recording date and timecode — the open
question gating task 2.8 (`JoinMetadata` stamping in `processConcatenateJob`, which currently
stamps nothing). Triggered by the user asking whether we'd ever settled TC correction / writing a
TC track into the output, and noting that **Penumbra** already has manual date/TC entry.

## What happened

### Surfaced the contradiction that stalled 2.8
Two 2026-06-07 decisions quietly conflict:
- *"Timecode is authoritative for the metadata fix"* — trust the source start-TC, rewrite the date
  to match.
- The research-revised grouping decision — DJI's embedded `tmcd` is **almost always `00:00:00:00`**
  and `creation_time` is **frequently wrong** (1904/1951 epoch + TZ bug).
So "trust the source TC" had nothing real to trust. That's why 2.8 was never wired — the right
behaviour was genuinely undecided, not just unimplemented. Confirmed against the code: the
`JoinMetadata` struct, the FFmpeg `-timecode`/`-metadata creation_time` args, and the
~300-line `QuickTimeAtomWriter` all exist and are plumbed, but `processConcatenateJob` passes no
metadata (explicit `TODO (task 2.8)`).

### Mined Penumbra for the proven pattern (the user's pointer)
Read `~/ProgrammingProjects/1-macOS/Penumbra`'s timecode subsystem — it ships exactly this:
- `Utils/DateCorrectionResolver.swift` — `Resolution{date, provenance, mismatchDetected}` from a
  **source-priority chain** (Penumbra's order: filename-parsed date → filesystem `creationDate` →
  unavailable; IN-point offset added).
- `Utils/TimecodeFormatter.swift` — `wallClockTimecode(for:frameRate:isDropFrame:)` derives the
  `-timecode HH:MM:SS:FF` string from a `Date` (Calendar.current TZ, floor-not-round on frame,
  throws on drop-frame at non-29.97/59.94); `ISO8601Z.format` for `creation_time` /
  `com.apple.quicktime.creationdate`.
- `Utils/SourceTimecodeReader.swift` — TN2310 `tmcd` reader (detects a non-zero source TC).
- **Manual entry** = per-item `dateOverride: Date?` with `effectiveDate = dateOverride ?? resolvedDate.date`
  (`Models/ExportDialogModel.swift`) + an "Override…" popover (`Views/ExportDialogView.swift`).
- `ExportManager.swift` — assembly + a **load-bearing FFmpeg gotcha**: when emitting `-timecode`
  you MUST drop the source data/`tmcd` track (`-map -0:d`), else `-c copy` passes the old `tmcd`
  through and FFmpeg silently ignores the new `-timecode`. Conjoyn's join already does `-map -0:d`,
  so it's covered.

## Decisions
- **Resolved-wall-clock is authoritative; both the date atoms and the TC track derive from it;
  manual override is the final fallback.** Supersedes (reframes, doesn't delete) the 2026-06-07
  "TC authoritative" entry. Resolution priority tuned for DJI: **SRT first-cue wall-clock →
  filename datetime → sane `creation_time` → filesystem date → manual override**; source `tmcd`
  honoured only if non-zero. Full entry + Penumbra port-source map in `docs/decisions.md`
  (2026-06-09, top).

# Session 7 — Implement + validate task 2.8 (date + start-timecode stamping)

## Focus
Build the s6-decided model and prove it on the real card: stamp the joined file's `creation_time`
and `tmcd` start timecode from one resolved recording-start wall-clock.

## What happened

### `RecordingStartResolver` (new) + Penumbra formatter ports
New `Services/RecordingStartResolver.swift` resolves one authoritative recording-start `Date` per
record group from a DJI-first chain, **manual override → SRT first-cue → filename datetime → sane
embedded `creation_time` → filesystem date**:
- Zone-free wall-clock signals (SRT, filename) are interpreted in `Calendar.current` (the operator's
  machine); the embedded/filesystem signals are already absolute and pass a **sanity gate**
  (≥ 2010-01-01, ≤ now+1d) that rejects the 1904/1951 QuickTime-epoch artifacts and far-future clocks.
- A **timezone-safe mismatch flag** compares only the two wall-clock signals (SRT vs filename) —
  comparing a wall-clock against the absolute embedded date would falsely fire on a legitimate UTC
  offset, so that's deliberately skipped. Threshold 120 s.
- Ported `TimecodeFormatter.wallClockTimecode(for:frameRate:isDropFrame:)` and `ISO8601Z` from
  Penumbra verbatim (DJI records non-drop-frame → `isDropFrame: false`). The Swift-6 strict-
  concurrency build rejected the shared `ISO8601DateFormatter` static; marked `nonisolated(unsafe)`
  (configured once, read-only thereafter).

### Wiring (`QueueManager.resolveJoinMetadata`)
Replaced the no-op TODO in `processConcatenateJob`. The new helper resolves the first segment, then
derives **both** outputs from the one `Date`: `creation_time` via `ISO8601Z.format`, start timecode
via `wallClockTimecode` at segment-1's probed fps (fallback 30). Honours the per-job toggles
(`fixCreationDate` gates the date, `preserveTimecode` gates the timecode) and a new
`ConversionSettings.dateOverride` (model + plumbing now; override UI is Wave 6). Logs the chosen
provenance and any mismatch. Empty metadata (toggles off / nothing resolves) stamps nothing — the
lossless `-c copy` join and param guard are untouched; FFmpeg writes the tags during the mux and
`-map -0:d` clears DJI's data tracks so `-timecode` is not silently ignored.

### Real-footage validation (card `DJI_001`)
Ran the **exact production arg vector** with the bundled LGPL ffmpeg on the real split tail
`0008+0009` (each a real DJI container: `hvc1` + `djmd` + `dbgi` data tracks + `mjpeg` preview).
The first segment's SRT first cue is `2026-05-21 19:53:03.448` (CEST, Berlin — lat 52.67/lon 13.67),
and the source `creation_time` is `17:53:03Z` (the same instant). Resolver picked the SRT cue →
output stamped:
- `creation_time = 2026-05-21T17:53:03Z` (mvhd stores whole seconds; the .448 sub-second is dropped
  by the container — instant matches the source exactly), and
- `timecode = 19:53:03:11` (local 19:53:03 + frame `floor(0.448 × 25) = 11`).
Output streams = **one `hvc1` video + a fresh `tmcd`**; the `djmd`/`dbgi`/`mjpeg` tracks are gone.
Confirms `-map 0:v:0` + `-map -0:d` + `-timecode` all behave on real DJI footage.
